# evo-talos Devkit v0.2 Release Notes

Compared with: [`v0.1`](https://github.com/evolplus/talos/tree/v0.1)  
Release date: 2026-06-20  
Scope reviewed: current plugin worktree against the local `v0.1` tag

## Summary

v0.2 makes evo-talos more usable as a cross-assistant SDLC kit and tightens the Figma-to-implementation path. The release adds Codex-native command-shim skills, improves project initialization and SDLC loop entrypoints, strengthens Figma design extraction and canvas layout rules, and introduces a Design Element Manifest so FE Dev must implement the actual fields, labels, items, actions, states, and copy from the approved design.

Product diff against `v0.1`, excluding this release-note file:

- 27 files changed
- 735 insertions
- 182 deletions
- 2 new SDLC command-shim skills

This release also adds this release-note document under `docs/releases/`.

## Highlights

- Added Codex-compatible `sdlc-init` and `sdlc-loop` skill entrypoints.
- Updated Claude, Codex, and marketplace metadata for v0.2 and fixed the Claude manifest display-name schema.
- Improved Flow A Figma requirement extraction so design guidelines can be inferred from existing designs.
- Added deterministic Figma canvas layout guidance to prevent overlapping screen frames.
- Added a Design Element Manifest and related gates to reduce FE implementation drift from Figma.
- Updated README and usage documentation with clearer Claude Code and Codex workflows.

## Added

### Codex SDLC Entrypoints

Codex now has command-shim skills equivalent to the Claude Code slash commands:

- `skills/sdlc-init/SKILL.md`
- `skills/sdlc-init/agents/openai.yaml`
- `skills/sdlc-loop/SKILL.md`
- `skills/sdlc-loop/agents/openai.yaml`

This allows Codex users to initialize and run the kit with prompts such as:

- `Run sdlc-init`
- `Run sdlc-init --dry-run`
- `Start the SDLC loop`
- `Run sdlc-loop`
- `Start project`

### Design Element Manifest

UI/UX handoffs now require a `Design Element Manifest` that enumerates every implementation-relevant Figma element:

- Required fields and form controls
- Labels, placeholders, helper text, validation copy, and state copy
- List/table/card items and repeatable content
- Buttons, links, tabs, menus, toggles, and action controls
- Empty, loading, error, disabled, selected, and success states
- Decorative exclusions that FE Dev may ignore

The manifest gives FE Dev a concrete checklist instead of relying on visual approximation alone.

### Visual-Spec Assertions

QA visual specs now include `Design Element Assertions` derived from the handoff manifest. These assertions help QA verify that the built UI includes the required design elements, not just a similar layout.

### FE Traceability

FE Dev references now require:

- Frozen Figma refs
- A `Design Element Manifest`
- An `Implementation Trace Matrix`

The trace matrix maps each required design element to the concrete source implementation and evidence, making omissions easier to catch before QA.

## Changed

### Project Initialization

`sdlc-init` guidance now distinguishes Claude Code and Codex targets:

- Codex target writes `AGENTS.md` and avoids injecting Claude-specific `.claude` settings by default.
- Claude target writes `CLAUDE.md`, merges safer Claude settings, and installs hook scripts.
- Both-target mode writes both instruction files and applies Claude-specific settings/hooks.

The command is also documented as the way to refresh project-local injected guidance after plugin updates.

### Plugin Metadata

- Claude plugin manifest now uses `displayName`, matching the strict plugin schema.
- Claude plugin manifest version is `0.2`.
- Claude marketplace entry version is `0.2`.
- Codex plugin manifest version is `0.2.0`.
- Codex plugin default prompts now point users toward `sdlc-init`, `sdlc-loop`, and brownfield onboarding.

### Figma Requirement Extraction

Flow A Figma extraction now captures design-guideline evidence when the design itself is the source of truth:

- Color palette
- Typography
- Spacing scale
- Border radius
- Component and state patterns
- Design-system signals

When no separate design guideline exists, BA can record `Design-Guideline: from-figma` in the SRS.

### Figma Canvas Layout

The UI/UX Designer workflow now includes stronger canvas layout rules:

- Resolve the scoped Figma page before reading or writing.
- Place screen frames on a deterministic grid.
- Keep screen frames non-overlapping.
- Run a pre-handoff overlap lint.
- Treat overlapping screen frames as a handoff blocker unless explicitly justified.

This improves reviewability in Figma while preserving per-screen development handoff.

### FE Framework Standard

FE Dev guidance now reinforces framework selection from the SRS and brownfield evidence:

- React Native
- ReactJS
- Flutter
- Vue.js
- Angular
- Next.js

The FE standard also emphasizes exact design-element implementation from confirmed Figma refs.

### BA Verification

BA design completeness and post-implementation checks now account for the Design Element Manifest:

- Phase 3 verifies that the design handoff includes manifest coverage.
- Phase 5 checks that FE output implements the required manifest rows before a UI task can progress.

### Runtime Guards

Two UI-related hooks were strengthened:

- `hooks/fe-dev-design-contract-guard.cjs`
- `hooks/ui-task-readiness-guard.cjs`

The guards now block unsafe progression when required design refs, manifests, trace matrices, or visual-spec assertions are missing.

### Documentation

README and usage documentation were expanded to describe:

- Claude Code and Codex setup paths
- Token-consumption expectations
- Greenfield mode usage
- Brownfield onboarding
- FE and BE framework tracks
- Figma-first design flows
- Design Element Manifest expectations

## Fixed

- Fixed Claude plugin manifest display-name schema compatibility by using `displayName`.
- Reduced Figma canvas overlap risk by adding explicit layout and lint guidance.
- Reduced FE implementation drift by requiring a manifest-backed design contract.
- Reduced ambiguity in Flow A by extracting design guidelines from Figma when no standalone guideline exists.

## Compatibility And Upgrade Notes

- Projects initialized with v0.1 should re-run initialization to receive updated injected guidance:
  - Claude Code: `/sdlc-init`
  - Codex: `Run sdlc-init`
- In-flight UI tasks created before v0.2 may need refreshed UI/UX handoffs, visual specs, and FE refs so they include:
  - `Design Element Manifest`
  - `Design Element Assertions`
  - `Implementation Trace Matrix`
- v0.2 introduces stricter UI gates. Some tasks that previously passed with approximate design implementation may now be blocked until missing Figma elements are implemented or explicitly marked decorative/out-of-scope.
- Codex users should start a fresh thread after installing or updating the plugin so the new skill list is loaded by the runtime.
