# Hard Rules — Operational Detail

This file holds the role-specific + situational Hard Rules that were previously embedded in CLAUDE.md §10. Section number is preserved for cross-reference compatibility.

For workflow contract entry-point, see `CLAUDE.md`. CLAUDE.md §10 keeps the ~8 **universal** Hard Rules every agent must internalize on every dispatch; this file holds the role-specific + skill-specific + hook-enforced rules that loaded only when relevant.

Many of these rules are ALSO declared in their owning agent's template Hard Rules section (FE Dev's contract-frozen rule lives in `_templates/fe-dev.md`; DevOps's project-scoped container rule lives in `_templates/devops.md`; etc.). This file is the centralized index that the Orchestrator + Agent Generator consult when they need the broader view; per-role agents typically read only their own template.

---

## 10a. Role-Specific + Hook-Enforced Hard Rules

### Source-code + dispatch rules

- **Orchestrator does not write source code directly.** Code that ships — anything under `**/src/`, `e2e/**/*.spec.*` files, and project-declared custom source directories — is the domain of BE Dev / FE Dev (Path A) or the Path B2 debugger trivial-fix exemption flow (which still routes through Dev dispatch, not direct Orchestrator write). The Orchestrator's "I'll just fix this one line" instinct silently bypasses the SDLC pipeline: no BA spec update, no SA architecture cross-check, no TL task entry, no QA-Author test case, no Code Reviewer lens. The `source-code-write-guard.cjs` hook refuses at runtime when the proposed Write/Edit/MultiEdit/NotebookEdit targets a source-code path **outside any `.worktrees/<role>-<task-id>/` directory** — i.e., the Orchestrator's main-repo cwd. Sub-agents always write inside their own worktree per `.claude/rules/worktree-isolation.md` §5, so their writes fall under the `.worktrees/` exclusion segment and pass. **The gate is block-by-default — no env var required to engage it.** Escape hatch `CLAUDE_ALLOW_ORCHESTRATOR_CODE=1` permits trivial Path D fixes (one-liner typos, kit dogfooding) with rationale documented in SRS §10 Changelog; use sparingly. See `.claude/rules/task-type-routing.md` §11 for Path A (SDLC) vs Path B2 (trivial fix via Dev) vs Path D (non-code only).

- **Orchestrator does not perform sub-agent work, even when dispatch fails.** The Orchestrator coordinates; it never executes any sub-agent's procedure directly via Read/Edit/Write — not BA's SRS ingestion, not SA's architecture authoring, not TL's task breakdown, not any role's work. This applies *especially* when sub-agent dispatch fails (file missing, frontmatter wrong, runtime error, role file lacks `name:` matching the `subagent_type:`, etc.). On dispatch failure, the Orchestrator MUST: (1) halt the current flow; (2) report the exact failure to the operator; (3) ask for guidance and wait. The "Pros: kickoff continues now / Cons: violates §10" rationalization is explicitly rejected by the kit. Doing BA's work directly may produce a working SRS, but it produces a SRS that wasn't authored by an agent operating under BA's mode procedures — so downstream agents inherit silent gaps that surface as integration errors weeks later. This rule has NO escape hatch. The resolution is to fix the agent definition (add frontmatter, add the missing file, run Agent Generator in `bootstrap` mode) — never to substitute the Orchestrator.

- **Role-specialized dispatch required.** When dispatching any work that has a kit-defined role, the Orchestrator MUST use the corresponding specialized sub-agent via Claude Code's `subagent_type` parameter. Mapping (BA → `ba`, UI/UX Designer → `ui-ux-designer`, SA → `sa`, TL → `tl`, QA-Author → `qa-author`, BE Dev → `be-dev`, FE Dev → `fe-dev`, DevOps → `devops`, QA-Exec → `qa-exec`, Agent Generator → `agent-generator`, Researcher → `researcher` Path B1, Debugger → `debugger` Path B2, Code Reviewer → `code-reviewer` Path B3, OQ Resolver → `oq-resolver` Path B4, Codebase Archaeologist → `codebase-archaeologist` Path B5). **`subagent_type: general-purpose` is FORBIDDEN** for any work that has a kit-defined role. Falling back to general-purpose silently strips the kit's hard rules, mode procedures, and tool scope from the dispatch. If the role-specific agent file is absent, the Orchestrator MUST dispatch the Agent Generator first (in `bootstrap` mode for pre-sign-off dispatches; default mode otherwise). Escape hatch: `CLAUDE_ALLOW_GENERAL_PURPOSE=1` env var permits general-purpose for one-off cross-cutting work that genuinely has no kit role; document rationale in SRS §10 Changelog. See `.claude/rules/orchestrator-operating-rules.md` §9 Step 4.5.

### Source layout discipline

- **All shipping application source lives under two fixed roots: `frontend/` (FE Dev) and `backend/` (BE Dev).** Declared per project in SRS §3.4.5 Source Layout. FE Dev writes UI / client / frontend-build source ONLY under `frontend/`; BE Dev writes server / data-layer / job / integration source ONLY under `backend/`. Single app/service per tier → source directly under the root (`frontend/src/**`, `backend/src/**`); multiple → one sub-directory per app/service (`frontend/<app-slug>/**`, `backend/<service-slug>/**`, slugs mirroring `docs/architecture.md` C4 container names). Source that is genuinely neither tier (shared library, codegen output) requires an explicit extra root declared in SRS §3.4.5 (`<tier> root: <path>`) or passed via `CLAUDE_SOURCE_CODE_DIRS`. Test code (`e2e/`, `integration_test/`, `.maestro/`, platform UITest dirs) is QA-owned and lives outside these roots. The `source-code-write-guard.cjs` hook enforces at runtime — it strips the worktree prefix and refuses any `/src/` source write whose worktree-relative path is not under a declared root. The hook cannot distinguish FE from BE (no role identity in the PreToolUse event), so it enforces "all source under a declared root"; the FE→`frontend/` / BE→`backend/` binding is the prose discipline carried in the be-dev / fe-dev templates. Escape hatches: `CLAUDE_SKIP_SOURCE_LAYOUT_CHECK=1` (operator-explicit; legacy layouts / one-off cross-tier work), `CLAUDE_SOURCE_CODE_DIRS=<prefixes>` (extra roots). Document rationale in SRS §10 Changelog. See [`.claude/hooks/lib/source-layout.cjs`](.claude/hooks/lib/source-layout.cjs) + SRS §3.4.5.

- **Frontend framework comes from SRS, not FE Dev preference.** SRS header `Frontend-Framework:` is mandatory for any frontend/UI project and must be one of `React Native`, `ReactJS`, `Flutter`, `Vue.js`, `Angular`, `Next.js`, `multiple`, or `N/A`. Greenfield/authored SRSs get this from requirements or a BA `NEEDS_CONTEXT` choice before sign-off. Brownfield SRSs get it from code evidence during BA Mode E and record per-surface/per-app mappings in §3.4.2 / §3.4.5. FE Dev loads `.claude/skills/fe-framework-coding-standard/` based on this SRS value; package-file/source detection at implementation time is only a drift check. Missing / `TBD` / unsupported / ambiguous framework blocks FE dispatch.

- **Backend track/framework come from SRS, not BE Dev preference.** SRS headers `Backend-Track:` and `Backend-Framework:` are mandatory for any backend/server-side project. `Backend-Track:` must be `backend-web`, `backend-service`, `multiple`, or `N/A`. `Backend-Framework:` must be `TypeScript with Express`, `TypeScript with NestJS`, `Python with FastAPI`, `Java with Spring Boot`, `.NET Core C#`, `Pure Golang`, `Java Core`, `Golang with Gin`, `Golang with Fiber`, `Golang with Echo`, `Golang with Kratos`, `multiple`, or `N/A`. Greenfield/authored SRSs get these from requirements or a BA `NEEDS_CONTEXT` choice before sign-off. Brownfield SRSs get them from code evidence during BA Mode E and record per-service mappings in §3.4.5. BE Dev loads `.claude/skills/be-framework-coding-standard/` based on these SRS values; manifest/source detection at implementation time is only a drift check. Missing / `TBD` / unsupported / ambiguous backend track/framework blocks BE dispatch.

### Workflow gate rules

- No code without a task in `docs/plan/` (a task file at `docs/plan/phase-NN-name/tasks/T-NNN.md`)
- No task without a linked User Story (US-NNN) — and where applicable, a linked FR (FR-NNN) — in `docs/SRS.md`
- BA never modifies upstream SRS functional content; augmentations are additive only and live in clearly-labeled engineering sections
- BA does not block SRS sign-off on missing designs. Design existence is verified post-sign-off by BA Phase 3 against the UI/UX Designer's handoff.
- **No FE Dev start on a backend-dependent task until the API contract is `Frozen`**
- **FE Dev must not write to `docs/api-contracts/`**
- No FE Dev UI implementation until the task's design sub-status = `design-confirmed`
- The Designated Design Approver is the only authority that can transition a task's design sub-status to `design-confirmed`. Agents cannot self-confirm.

### Path-A SDLC purpose gate

- No Path A SDLC dispatch (BA → SA → TL → Dev → DevOps → QA-Exec) against a SRS whose header carries `Purpose: documentation`. Documentation-purpose SRSs are reference artifacts only; promoting to governance requires an explicit re-confirmation pass (`.claude/rules/brownfield-onboarding.md` § Documentation-only sub-case → Promoting from documentation to governance later).

### Sign-off authority (mirror of `srs-signoff-protocol.md` Hard Rule)

- Only `srs-source-validator` may transition `Ready-for-Sign-off → Source-Validated`.
- Only `srs-feasibility-validator` may transition `Source-Validated → Signed-off`.
- BA cannot self-sign-off (caps at `Ready-for-Sign-off`).
- The Orchestrator cannot manually flip any sign-off Status transition.
- The operator cannot override (escape hatches do not apply to these transitions).

### Architecture validation authority (mirror of `sub-agent-registry.md` §3.11)

- Only `architecture-validator` may transition `docs/architecture.md` `Status: Draft → Validated`.
- SA cannot self-validate its own architecture (the kit's author-≠-approver discipline — same reason BA cannot self-sign-off the SRS).
- **TL must not be dispatched while architecture `Status ≠ Validated`** (the design-side analogue of "no downstream work while SRS Status ≠ Signed-off"). The Orchestrator enforces the sequencing at `.claude/rules/orchestrator-operating-rules.md` §9 Step 3.7; `architecture-status-guard.cjs` surfaces a reminder when the gate is closed.
- The Orchestrator cannot manually flip the transition; the operator cannot override (no escape hatch). On `unqualified`, architecture stays `Draft` and SA is re-dispatched against the validator's revision list.

### Open-issues triage gate

- No new dispatch while any `docs/open-issues.md` entry is in `open` state.

### Self-containment invariant

- **Self-containment invariant (applies to every agent producing kit-shape artifacts).** Kit artifacts (`docs/SRS.md`, `docs/user-stories/<US-ID>.md`, `docs/frs/<FR-ID>.md`, `docs/external-integrations/<system-slug>.md`, `docs/architecture.md`, `docs/decisions/<ADR-NNNN>.md`, `docs/test-cases/by-us/<US-ID>/*.md`, `docs/test-cases/by-task/<task-id>/*.md`, `docs/uiux/handoffs/<task-id>.md`, `docs/uiux/visual-specs/<task-id>.md`) are **self-contained for engineering use**. Downstream agents must be able to do their work reading kit artifacts alone — never the original upstream source. Allowed: audit-annotation lines (`Synthesized-From:`, `Source:`, `Source-Hash:`, `Source-Last-Pulled:`, `Figma-File-URL:`, `Figma-File-Version:`, `Last-Confirmed:`), lateral kit references (`Linked FRs:`, `Linked Component:`, `Linked SRS:`, `Linked anchor:`, `Refs: ADR-NNNN`), and changelog mentions of historical ingestion. Forbidden: substantive content references back to upstream input in body content — phrases like `see docs/requirements/<X>`, `see docs/archaeology-reports/<X>`, `refer to <Confluence URL>`, `details in <upstream path>`. If the upstream fragment carried content the kit artifact needs, **inline that content**; if it's missing, leave a `TODO: <field expected>` marker + raise an OQ per the no-invention invariant. Never offload to upstream as a substantive reference. The `self-containment-validator.cjs` hook enforces this at write time; BA Phase 1.X common procedure runs a self-containment grep check.

### BA Mode D verbatim capture

- **BA Mode D captures user requirements VERBATIM before synthesis.** When the operator gives BA a new requirement mid-project, BA's FIRST action is to write `docs/requirements/conversational-additions/<ISO-date>-<slug>.md` containing the operator's input verbatim (with operator name, timestamp, ingestion context). THEN BA proceeds with normal Mode D synthesis. This keeps `docs/requirements/` the complete audit log of every requirement that ever entered the project — the srs-source-validator's coverage check operates against a complete source corpus, not just initial PM input. See `.claude/agents/_templates/ba.md` § Mode D capture step.

### API contract format declaration

- **API contract format declared in SRS §3.4.4.** Every project with any API surface (REST / gRPC / GraphQL / async messaging) MUST declare the contract format per API style in `docs/SRS.md` §3.4.4. Defaults: `openapi-3.1` for REST, `proto3` for gRPC, `graphql-sdl` for GraphQL, `asyncapi-2.x` for messaging. Deviations (including markdown contracts) require an ADR reference in the Justification column. BE Dev's `api-contract-author` skill READS this declaration; per-task format choice is forbidden. SRS §3.4.4 missing for an API-bearing project blocks sign-off (raised as OQ category `api-contract-format-undeclared` by BA Phase 1.X). See [`.claude/skills/api-contract-author/SKILL.md`](.claude/skills/api-contract-author/SKILL.md).

### Third-party dependency Approver

- **No new third-party dependency** (paid service, OSS library, managed cloud service, external API, first-time-use self-hosted vendor tool) enters `docs/architecture.md` or `docs/decisions/` without an ADR carrying an explicit human `Approver` field (named in SRS's `Designated Dependency Approver`). SA halts and requests confirmation via `NEEDS_CONTEXT` when proposing a new dependency. Prior-ADR approvals for the same vendor on the same project propagate — cite via `Refs: ADR-NNNN` and skip re-approval. See `.claude/skills/third-party-dependency-evaluation/SKILL.md`.

### External-integration adequacy gate

- **External-integration adequacy gate (strict, blocks SRS sign-off).** Every external system the product touches — including in-org defaults (Account/Passport, Kafka), managed cloud services, and third-party APIs — requires a paired `docs/external-integrations/<system-slug>.md` file at `Adequacy: adequate` (or `Adequacy: deprecated` — see pass-through note below) before BA may flip SRS Status to `Ready-for-Sign-off`. BA identifies external systems during Phase 1.X step 9 and files placeholders at `Adequacy: inadequate`. The Orchestrator dispatches SA in `external-integration-adequacy` mode against the placeholders; SA fills per-operation interface detail (endpoint, auth, schemas, errors, rate limits, retry, idempotency, NFR posture, failure modes) and is the ONLY agent permitted to flip `Adequacy: adequate`. SRS §3.5 indexes every integration; coverage is strict (in-org defaults are NOT exempt; solution-defaults pre-fills but doesn't substitute). `Adequacy: deferred` does NOT pass the gate (tracking value only). **Pass-through states:** `adequate` and `deprecated` both pass; `inadequate`, `deferred`, and missing-field all block. `Adequacy: deprecated` marks an integration that is RETIRED but whose file is retained for audit per SRS §3.5 — it has no active consumer, so the gate (whose purpose is to guarantee a downstream dev can build against an ACTIVE integration) does not apply. SA is the only agent permitted to flip `Adequacy: deprecated` (same authority as `adequate`), recording the transition in the file's §8 Changelog. Provenance guard: a `deprecated` file MUST also be marked deprecated in the SRS §3.5 index; BA Phase 2 step 3 cross-checks this so `deprecated` cannot be used to skip adequacy work on a still-live system. The `external-integration-adequacy-validator.cjs` hook enforces at SRS write time; BA Phase 2 step 3 runs the same check as a final audit. See [`.claude/agents/_templates/_artifacts/external-integration-template.md`](.claude/agents/_templates/_artifacts/external-integration-template.md).

### Figma + design-flow rules

- Human edits made directly in Figma between handoff and confirmation must be absorbed via `design-human-edited` → `incorporate` mode. The Orchestrator never silently accepts a different Figma version than what the most recent handoff describes; the handoff must always match the version under review.
- Figma file version ID at `design-confirmed` is immutable for the task. If Figma changes, treat as blocking issue per `.claude/rules/change-synchronization.md` §7.
- UI/UX Designer is the only agent permitted to write Figma via MCP. FE Dev and QA-Author remain read-only.
- **Figma page-scoping is mandatory (applies to every Figma-reading operation).** A Figma file has multiple pages (top-level tabs); the project's design lives on ONE specific page. The SRS §3.4.1 header field `Figma-Design-Page-Node-ID` (paired with `Figma-File-URL`) names that page. Every Figma-reading agent (UI/UX Designer in `extract` / `map` / `create` / `import` / `revise` / `incorporate` modes; FE Dev when pulling tokens for the design contract; QA-Author when authoring visual specs) MUST resolve this field at Step 0 of every dispatch and scope all subsequent reads to the subtree under the resolved page node. Frames on other pages are OUT OF SCOPE. Cross-page Node IDs in any kit artifact (handoff, refs, visual-spec, mapping) are a `figma-cross-page-reference` violation. BA Phase 1.X step 10 captures the page Node ID from the URL's `?node-id=` parameter OR halts with NEEDS_CONTEXT asking the operator. Without this discipline, Figma-reading operations walk every page (mixing brainstorms, old designs, design-system Foundation, the project design) — either grabbing irrelevant frames or missing the project design entirely.
- **Design-Flow detection is mandatory pre-sign-off when SRS has UI surfaces.** BA Phase 1.X step 10 scans the requirements for Figma URLs and sets SRS header `Design-Flow:`:
  - `A` (Figma provided) → BA dispatches UI/UX Designer in `map` mode pre-sign-off; mapping artifact at `docs/uiux/figma-mappings/v<srs-version>.md` qualifies the design at sign-off time. BA Phase 2 step 3.5 blocks `Status: Signed-off` until `Mapping-Status:` is `qualified` (or `orphans-only` with PM disposition).
  - `B` (no Figma, agent designs fully) or `C` (no Figma, agent + human modifies) — chosen by user via BA NEEDS_CONTEXT. SRS sign-off blocked until the choice is recorded.
  - `N/A` — SRS has no UI surface; design phase skipped entirely.
- **UI/UX Designer `create` mode requires explicit Design-Flow B or C.** Never runs by default. When `Design-Flow: A` the kit short-circuits the post-sign-off design lifecycle entirely — designs are already qualified via `map`. If `map` returns `Mapping-Status: gaps`, the resolution is PM-side (add screens to Figma OR reduce SRS scope) — NOT auto-`create`. See `.claude/rules/parallel-execution.md` §4 for the Design-Flow A/B/C matrix and `.claude/skills/figma-srs-mapping/SKILL.md` for the `map` procedure.

### QA-Exec discipline

- QA-Exec must run Tier 1 + Tier 2 for all UI test cases; Tier 3 only when SRS marks the requirement `Visual-Critical: yes`
- No QA-Exec UI run against a stale visual spec (Figma version mismatch)
- No task closure with open coverage gaps in the visual spec
- No `done` status without QA execution pass

### Project state rules

- No SRS change without BA action and downstream re-sync
- No sub-agent writes directly to anything under `docs/plan/` — proposals only via `plan-update.json`

### Git + commit discipline

- Every commit on any branch follows `.claude/skills/git-commit/SKILL.md` — identity configured, conventional message format with task traceability (`Refs: T-NNN` or in-subject `(T-NNN)`), no secrets, no PII, attribution trailers where applicable.
- **Pre-flight git setup is mandatory.** The Orchestrator runs `git init` (if needed) + committer-identity check at every invocation per `.claude/rules/orchestrator-operating-rules.md` §9 Step 0. Without a working git context, sub-agent commits fail and the "commit before signaling done" discipline collapses. Identity unset → halt with `NEEDS_CONTEXT`; never auto-configure from environment guesses.
- **Sub-agents commit before signaling done.** Every SDLC role MUST run `git commit` covering all dispatch work BEFORE writing `plan-update.json` (the dispatch-completion signal). The `task-completion-commit-check.cjs` hook refuses `plan-update.json` writes when `git status --porcelain` is non-empty. Non-SDLC roles MUST commit before returning to the Orchestrator (prose rule; Orchestrator validates at return-time). Agent Generator MUST commit generated agent files before returning. Intermediate commits during a dispatch are encouraged; the rule enforces only that the worktree is clean at the done-signaling moment. Escape hatch `CLAUDE_SKIP_COMMIT_CHECK=1` permits the write without committing — use sparingly, document rationale.

### DevOps + container discipline

- **Project-scoped container discipline (DevOps).** Every docker mutation (compose up/down/run/restart, plain container stop/rm/kill/restart, volume rm, network rm, image rm) operates only on containers belonging to the project's Compose project (identified by `COMPOSE_PROJECT_NAME` env / SRS project-name field / cwd basename, sanitized to lowercase + alphanumeric + dashes). Out-of-scope mutations are forbidden EVEN for cleanup — the operator's other local services (personal Postgres, sibling repos' stacks, unrelated containers) MUST remain untouched. Globally-destructive operations (`docker system prune`, `docker volume prune`, `docker network prune`, `docker container prune`, `docker image prune`, `docker rm -f $(docker ps -q)` variants) are unconditionally forbidden. On port conflict with an out-of-scope container, DevOps picks a different port via the probe procedure — never stops the other container. Read operations on out-of-scope containers (ps, inspect, logs, port, stats, network/volume ls) ARE permitted for conflict-detection. The `docker-scope-guard.cjs` hook enforces at runtime; the prose discipline lives in `.claude/skills/local-deployment/SKILL.md` § Project-scoped container discipline. Escape hatch `CLAUDE_SKIP_DOCKER_SCOPE_CHECK=1` — operator override only; document rationale.

### Data-contract rules

- **Format-boundary contracts (architecture.md §6).** Every data field that crosses two systems with different format specs for the same conceptual type (ISO-8601 vs MySQL DATETIME, integer-cents vs decimal-string, hex-UUID vs binary-UUID, JS Number vs BigInt for >2^53 values, locale-specific number formats, encoding variants, etc.) MUST be declared in `docs/architecture.md` §6 format-boundary rows with Field / Source format / Destination format / Boundary owner / Transformation function / Failure-mode classification. Source and destination formats specified EXACTLY (not "datetime" — "ISO-8601 with `T`+`Z`" vs "MySQL `DATETIME` zone-less"). BE Dev / FE Dev MUST invoke the named transformation function at the boundary, never pass values verbatim. Format violations are deterministic — §5 Failure Modes classifies them as `do-not-retry` so retry loops don't waste connections and bury the real driver error. QA-Author's format-boundary tests assert on the **converted value at the destination system**, NOT the source-system input. Integration tests against the local-deployment Docker stack run for every FR touching a §6 format-boundary row — mock DALs cannot enforce destination-system format constraints. Code Reviewer's `format-boundary` lens enforces at review time. See `.claude/skills/format-boundary-contracts/SKILL.md`.
- **Gate-field write ownership (architecture.md §6).** Every column written by one component and read by another to gate behavior (skip / state-machine / eligibility / idempotency / version / authorization) MUST be declared in `docs/architecture.md` §6 Cross-Component Data Contracts with column / owner(s) / write condition / consumers / gate semantics / explicit other-writer constraint. Agent Generator extracts §6 into BE Dev + FE Dev `## Project Specialization` as write-prohibitions. BE/FE Dev MUST NOT write a §6 column unless their task is named as an owner AND the write condition matches; the ORM-convenience pattern of stamping `updated_at`/`last_synced_at`/`touched_at` on every upsert MUST be suppressed for §6 columns. Code Reviewer's `data-lifecycle-contract` lens enforces at review time. See `.claude/skills/data-lifecycle-contracts/SKILL.md`.

### Design-implementation symmetry

- **Design-implementation symmetry (UI tasks).** The kit's design lifecycle is symmetric: design existence is gated rigorously (Designer handoff + BA Phase 3 completeness + Designated Design Approver confirmation + `design-confirmed` sub-status + immutable Figma version), AND implementation-matches-design is gated equivalently. For any task with a UI surface — `track: fe` / `be+fe`, or `Design sub-status:` set, or `Linked Surface:` non-null — the absence or content-incompleteness of ANY of the following artifacts is a `ready-for-deploy` closure-blocker, NOT a vacuous pass:
  - `docs/uiux/refs/<task-id>.md` with `Status: Frozen`, `## Design Element Manifest`, and `## Implementation Trace Matrix` (FE Dev's per-task design contract — produced before any UI source-code write)
  - `docs/uiux/visual-specs/<task-id>.md` with `## Design Element Assertions` (QA-Author by-task's component + field/item property contract)
  - `docs/test-cases/by-task/<task-id>/` with at least one TC file (QA-Author by-task's TC pack)
  - `docs/uiux/post-implementation-reports/<task-id>.md` with verdict `qualified` (BA Phase 5's design-vs-implementation completeness verdict — see `.claude/rules/parallel-execution.md` §4 Step 6)

  Three hooks enforce this at runtime: `fe-dev-design-contract-guard.cjs` refuses FE Dev source writes when the per-task design contract is absent, `Status: Draft`, or missing the Design Element Manifest / Implementation Trace Matrix. `ui-task-readiness-guard.cjs` refuses `plan-update.json` with `to_status: ready-for-deploy` when the artifact set is missing OR refs/visual-spec files lack the field/item-level manifest sections. (`task-completion-commit-check.cjs` continues to enforce the worktree-clean rule orthogonally.) Motivating incident: 2026-06-04 FR-022 batch-UI silent drop — T-168 closed `done` despite Scope A (batch UI) never being implemented; six layered gates between `design-confirmed` and Phase closure all returned vacuous PASS because their enforcement artifacts didn't exist. The current extension closes the related "approximate UI" gap: artifact presence alone is insufficient if the artifacts do not enumerate every Figma field/item/copy/action. The fix is uniform: artifact absence or content-incompleteness is a closure-blocker; UI-task closure REQUIRES the artifact set and manifest trace on disk before any agent can self-attest done; Phase exit criteria for UI-bearing FRs must mention UI-surface verification explicitly. Escape hatches `CLAUDE_SKIP_DESIGN_CONTRACT_CHECK=1` (non-UI FE tasks) and `CLAUDE_SKIP_UI_READINESS_CHECK=1` (operator-explicit override) — use sparingly and document rationale in the task file's Notes section.

### Worktree + handoff-artifact rules

- No two agents share a worktree
- **`plan-update.json` lives ONLY at `.worktrees/<role>-<task-id>/plan-update.json`.** The transient handoff artifact sub-agents emit to signal dispatch completion. Sub-agents write it inside their own worktree; the Orchestrator ingests its content into `docs/plan/` then cleans up the worktree per `.claude/rules/worktree-isolation.md` §5. Any `plan-update*.json` at project root, under `docs/`, or anywhere outside `.worktrees/<*>/` is leakage — not a kit-prescribed pattern, and refused at runtime by `plan-update-location-guard.cjs`. The Orchestrator's §9 Step 0.5 pre-flight sweeps any existing root-level stragglers on every invocation. Escape hatch `CLAUDE_ALLOW_PLAN_UPDATE_ROOT=1` permits one-off operator-explicit writes outside the worktree (e.g., kit-dogfooding); document rationale in SRS §10 Changelog.
- All kit-emitted artifacts live under `docs/`, organized in per-purpose subfolders. No agent writes kit-emitted artifacts to project root. Worktree root is reserved for transient handoff artifacts the Orchestrator consumes once and then discards (e.g., `plan-update.json`, the TL's `plan-proposal/` tree). Transient artifacts are NEVER git-merged to main; the Orchestrator ingests their content (e.g., copies `plan-proposal/` into `docs/plan/`) and cleans up the worktree per `.claude/rules/worktree-isolation.md` §5.
